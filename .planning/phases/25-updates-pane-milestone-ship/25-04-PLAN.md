## Trust Boundaries

|----------|-------------|

| prefs.json → pane | lastUpdateCheck/autoUpdateCheck are coerced upstream (Plan 02); the pane renders coerced values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-10 | Elevation of privilege | ungated pane | accept (by design) | D-25-1: the Updates pane is intentionally ungated (core infra, not a Pro feature). It exposes only version/check/toggle — no entitlement-bearing capability. No gate is the correct security posture here. |
| T-25-11 | Tampering (forged update via the pane) | Check → install | mitigate (reused) | The pane only triggers `runCheck`; install remains in the banner's `installUpdate` (minisign verify-before-apply unchanged, D-25-5). The pane cannot install an unverified update. |
| T-25-12 | Information disclosure | version display | accept | Version is public (About box, updater banner). No secret rendered. |
| T-25-13 | DoS (UI) | rapid Check clicks | mitigate | The button is disabled while `checking` is true; the shared singleton de-dupes in-flight state — no unbounded concurrent checks. |

No high-severity threats: ungated-by-design exposes only public metadata + a check trigger; install verification is unchanged and lives in the banner.
</threat_model>
