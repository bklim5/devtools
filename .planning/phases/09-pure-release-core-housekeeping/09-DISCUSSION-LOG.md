## Manifest edit approach

| Option | Description | Selected |
|--------|-------------|----------|
| Surgical for all 3 | Regex/string-replace only the version line per file; minimal diffs; Cargo scoped to `[package]`; only safe way to leave dep pins untouched | ✓ |
| JSON round-trip + Cargo regex | JSON.parse→stringify for the 2 JSON files (canonical but reformats/noisy diffs), regex for Cargo | |

**User's choice:** Surgical for all 3.
**Notes:** Captured robustness follow-up — regex must anchor to the first top-level `"version"` (JSON) / `[package].version` (Cargo) and fail loudly on 0 or >1 match.

## buildLatestJson API shape

| Option | Description | Selected |
|--------|-------------|----------|
| Single options object | `buildLatestJson({ version, pubDate, url, signature, notes? })` — named args, extensible | ✓ |
| Positional args | `buildLatestJson(version, pubDate, url, signature, notes?)` — terser, error-prone | |

**User's choice:** Single options object.
**Notes:** User first asked *why manifest.ts must be pure and what it's for*. Explained: it assembles the `latest.json` updater manifest; purity (no Date.now/fs) makes the corruption-prone dual-key assembly unit-assertable without mocks (success criterion #2), and keeps I/O in the Phase 11 driver which injects pubDate/notes/url/signature.

### notes field

| Option | Description | Selected |
|--------|-------------|----------|
| Optional, default `""` | notes defaults to empty when omitted; core stays unopinionated | ✓ |
| Required | Caller must always pass notes | |

**User's choice:** Optional, defaults to empty string.
