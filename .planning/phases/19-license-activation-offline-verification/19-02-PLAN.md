Machine file format (source-verified from keygen-api machine_checkout_service.rb):
- Outer: `-----BEGIN MACHINE FILE-----\n<base64 of envelope JSON, wrapped 80 cols>\n-----END MACHINE FILE-----`
- Envelope JSON: `{"enc": "<b64 dataset>", "sig": "<b64 64-byte ed25519 sig>", "alg": "base64+ed25519"}`
- Signature is over the ASCII string `"machine/" + enc` (the still-base64 enc with literal `machine/` prefix — Pitfall 4; signing the decoded bytes always fails)
- Dataset JSON: `{"meta": {"issued","expiry","ttl"}, "data": {<machine resource: id, attributes incl. fingerprint>}, "included": [<license>, <entitlements...>]}`

Verify algorithm (every failure -> typed error -> free tier; use `verify_strict`, never `verify`):
1. Missing file -> NotActivated (not an error state)
2. Strip BEGIN/END markers, remove all whitespace -> base64 decode -> envelope JSON (any failure -> Corrupt)
3. `alg != "base64+ed25519"` exactly -> UnsupportedAlg (encrypted "aes-256-gcm+ed25519" and anything else rejected — no weaker fallback)
4. b64-decode sig (64 bytes) -> Signature::from_slice (failure -> Corrupt)
5. `VerifyingKey::from_bytes(PUBKEY_32)` + `verify_strict(format!("machine/{enc}").as_bytes(), &sig)` -> failure = Tampered (a foreign CE instance's file also lands here — different account keypair)
6. b64-decode enc -> dataset JSON (failure -> Corrupt)
7. `data.attributes.fingerprint` != expected fingerprint -> ForeignMachine (ship-gate case 5)
8. Record `meta.expiry` in the result but do NOT enforce TTL expiry (Phase 21 — deferred per CONTEXT)
9. Success -> LicenseData { entitlement codes from included, expiry, issued }

Fingerprint (research-verified ioreg pattern from machineid-rs):
- `Command::new("ioreg").args(["-rd1", "-c", "IOPlatformExpertDevice"])` -> find line containing `IOPlatformUUID` -> split on `=` -> trim quotes -> uppercase hyphenated UUID string used VERBATIM as HMAC message (normalization locked — changing it later orphans every activation; document this in a code comment)
- `fingerprint = hex::encode(HMAC-SHA256(key = APP_SALT, message = uuid_string))` — lowercase hex, 64 chars

Status payload contract (shared with Plans 03/04 — define it HERE in mod.rs):
```rust
#[derive(serde::Serialize, Clone, Debug, PartialEq)]
#[serde(rename_all = "camelCase", tag = "state")]
pub enum LicenseStatusPayload {
    #[serde(rename_all = "camelCase")]
    NotActivated { has_stored_key: bool },
    #[serde(rename_all = "camelCase")]
    Licensed { expiry: Option<String>, entitlements: Vec<String> },
    #[serde(rename_all = "camelCase")]
    Problem { problem: ProblemKind, has_stored_key: bool },
}
// ProblemKind serializes as: "corrupt" | "tampered" | "foreignMachine" | "unsupportedAlg"
```
NOTE (LIC-04): no variant ever carries key material — `has_stored_key` is the ONLY Keychain-derived value JS may see. Keychain read errors map to `has_stored_key: false` (fail-soft for this boolean only; the verify path itself stays fail-closed).

|----------|-------------|
| machine.lic file -> verifier | Untrusted on-disk input (user-writable app data) crosses into Rust parse/verify |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-19-06 | T (Tampering) | verify.rs — forged/tampered machine.lic | mitigate | `ed25519_dalek::verify_strict` over literal `"machine/"+enc`; forgery requires the server-side private key which never leaves the CE DB; tests 2/5 prove rejection |
| T-19-07 | S (Spoofing) | machine.lic copied from another Mac | mitigate | Fingerprint HMAC compare in verify step 7 -> ForeignMachine; test 6 + ship-gate case 5 |
| T-19-08 | T | algorithm-confusion (encrypted/RSA cert fed to verifier) | mitigate | Exact `alg == "base64+ed25519"` string check, fail closed (UnsupportedAlg); test 4 |
