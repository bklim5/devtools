Base URL: `https://{config::KEYGEN_HOST}/v1/accounts/{config::KEYGEN_ACCOUNT_ID}`

Call 1 — validate (UNAUTHENTICATED; the key in the body is the credential):
`POST /licenses/actions/validate-key`
body: `{"meta": {"key": "<trimmed-key>", "scope": {"fingerprint": "<hmac-hex>"}}}`
Branch on `meta.code`, NEVER on `meta.valid` (Pitfall 3):
- `VALID` -> already activated on THIS machine -> skip to checkout (idempotent re-activation; the D-44 recovery path)
- `NO_MACHINE` | `NO_MACHINES` -> expected first-activation state -> proceed to call 2
- `FINGERPRINT_SCOPE_MISMATCH` -> seat taken on a DIFFERENT machine -> LicenseError::SeatLimit (D-36)
- `NOT_FOUND` -> LicenseError::InvalidKey
- `SUSPENDED` | `BANNED` -> LicenseError::Suspended; `EXPIRED` -> LicenseError::Suspended (calm terminal states)
Response also carries `data.id` (the license id) — capture it for call 2.

Call 2 — activate machine:
`POST /machines` header `Authorization: License <key>`
body: `{"data":{"type":"machines","attributes":{"fingerprint":"<hmac-hex>","platform":"macOS","name":"<hostname or 'Mac'>"},"relationships":{"license":{"data":{"type":"licenses","id":"<license-id>"}}}}}`
- 201 -> machine id
- 422 with error code `MACHINE_LIMIT_EXCEEDED` (exact string recorded in 19-SPIKE-OUTCOME.md — use THAT string) -> LicenseError::SeatLimit
- 403 -> configuration error (policy authenticationStrategy) -> LicenseError::ActivationFailed with a log line naming Pitfall 1

Call 3 — checkout machine file:
`POST /machines/{machine-id}/actions/check-out?include=license,license.entitlements&ttl=2629746` header `Authorization: License <key>` (NO encrypt param)
-> `data.attributes.certificate` (the `-----BEGIN MACHINE FILE-----` text)

Deactivate: `DELETE /machines/{machine-id}` header `Authorization: License <key>` -> 204

D-38 classification (in Rust, single source of truth): walk the reqwest error chain — OS-level network-down (`std::io::ErrorKind::NetworkUnreachable`/`HostUnreachable`-class causes, DNS failure with no network) -> `Offline`; connection-refused / timeout / TLS errors against a resolvable host -> `ServiceUnreachable`. If the chain is ambiguous, default to `ServiceUnreachable` (the more actionable message). (A4: exact introspection confirmed at impl time.)

Serialized error contract (Tauri command Err arm; thiserror + serde::Serialize as `{"code": "..."}` — copy lives in the webview, D-36/D-37/D-38):
`"seatLimit" | "offline" | "serviceUnreachable" | "invalidKey" | "suspended" | "activationFailed" | "noStoredKey" | "licenseProblem"`

TS contract added to src/lib/platform/index.ts (mirrors Plan 02's pinned serde JSON):
```ts
export type LicenseProblem = "corrupt" | "tampered" | "foreignMachine" | "unsupportedAlg";
export type LicenseStatusPayload =
  | { state: "notActivated"; hasStoredKey: boolean }
  | { state: "licensed"; expiry: string | null; entitlements: string[] }
  | { state: "problem"; problem: LicenseProblem; hasStoredKey: boolean };
export type LicenseErrorCode =
  | "seatLimit" | "offline" | "serviceUnreachable" | "invalidKey"
  | "suspended" | "activationFailed" | "noStoredKey" | "licenseProblem";

// Platform interface addition:
license: {
  status(): Promise<LicenseStatusPayload>;                       // pure-local, never network (D-45)
  activate(key: string | null): Promise<LicenseStatusPayload>;   // null => Rust uses the Keychain-stored key (D-44, LIC-04-safe)
  refresh(): Promise<LicenseStatusPayload>;                      // callable; UI wiring is Phase 21
  deactivate(): Promise<LicenseStatusPayload>;                   // callable-but-unwired (CONTEXT discretion)
};
```

Tauri async command rule (documented): async commands borrowing `State<'_, T>` MUST return `Result`. Pattern:
```rust
pub struct LicenseState(pub tokio::sync::Mutex<LicenseManager>);
#[tauri::command]
async fn activate_license(state: tauri::State<'_, LicenseState>, key: Option<String>)
    -> Result<LicenseStatusPayload, LicenseError> { ... }
```
App-defined commands registered via `generate_handler!` need no new capability entries (A3 — if invoke fails with a permissions error, add the one-line capability and note it in the SUMMARY).

Cargo.toml addition: `reqwest = { version = "0.13", default-features = false, features = ["json", "rustls-tls"] }` (rustls — no OpenSSL linkage). tokio comes via tauri; use `tauri::async_runtime`.

Release builds compile this path out entirely — no env var can weaken release TLS.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| webview -> Rust commands | Untrusted JS input (key string) crosses into Rust |
| Rust -> Keygen CE | Network call over TLS; server responses are untrusted input |
| Rust -> webview | Status payloads / error codes only |
| debug build -> release build | Dev-only CA trust must not survive into release |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-14 | I (Info disclosure) | key material crossing to JS | mitigate | Commands return LicenseStatusPayload/`{code}` errors only; key enters Rust once via activate(key) and never returns; acceptance greps + state-machine tests prove no key field in payloads |
| T-19-15 | T (Tampering) | MITM on activation | mitigate | reqwest rustls with default cert validation; checkout response is additionally verified through verify.rs (Ed25519) before any write — a MITM'd cert fails local verification |
| T-19-16 | T | dev CA trust weakening release TLS | mitigate | `DEVTOOLS_KEYGEN_CA` path compiled inside `#[cfg(debug_assertions)]` (webdriver double-gate precedent) — release builds contain no trust-injection code path |
| T-19-17 | S (Spoofing) | client-side seat decisions | mitigate | Zero client seat logic — Keygen's server-side validation codes are the only seat authority (Don't-Hand-Roll table); client only maps codes |
| T-19-18 | D (DoS) / fail-open | partial activation (machine created, checkout failed) leaving inconsistent state | mitigate | Write-after-verify ordering: machine.lic + Keychain written ONLY after a checkout cert passes local Ed25519 verification; test pins no-partial-write |
| T-19-19 | E (Elevation) | webview invoking privileged operations | mitigate | The 4 commands are the entire surface; no privileged Keygen (admin) tokens exist anywhere in the app (admin tokens live only in gitignored CE scripts); `license_status` is read-only local |
| T-19-20 | R (Repudiation) | silent error swallowing hiding activation failures | mitigate | Every failure path returns a typed code the UI must render (D-37); no `Ok` fallback arms |
</threat_model>
