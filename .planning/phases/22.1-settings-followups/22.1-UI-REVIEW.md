## Top 3 Priority Fixes

1. **Heading order jumps h3 → h2 inside the pane** (`LicenseSettings.tsx:298` h3 "License" → `:327`/`:380` h2 status; free state `:275` sr-only h3 → UpsellPanel `:436` h2 pitch) — a screen-reader user hears a level *increase* mid-pane, which is the classic axe `heading-order` smell even though it passes relative to the dialog's outer h2. The visible "License" header is the section title and should outrank the status banner. **Fix:** demote the status-banner `HEADING_CLASS` headings from `<h2>` to `<h3>` (or promote the section header logic) and migrate the `statusHeading()` e2e probe to read the first non-"Settings" heading by role+name rather than by `h2` tag, so the DOM order is h2(dialog) → h3(License) → h4(status). Removes the only structural a11y wrinkle in the phase.

2. **Form-only variant steals focus to the key input on mount, landing SR/keyboard users below the status card** (`UpsellPanel.tsx:231-237`, `focusOnMount` for `variant === "form-only"`) — when Settings opens directly on a problem/refreshNeeded License pane, focus jumps past the amber "License needs attention" banner straight into the field, so the user never hears why activation is needed. This is intentional per D-22.1-7 but is a genuine a11y nuance flagged in `deferred-items.md`. **Fix:** keep the field revealed but do NOT auto-focus on direct-open of the pane; reserve the focus move for the in-session transition into the problem state (where the user already has context). Confirm via the existing problem-state e2e + a manual VoiceOver pass.

3. **Inline success-card "Done" button is a dead no-op** (`UpsellPanel.tsx:381` `onClick={() => onDismiss?.()}`, `InlineActivation` rendered without `onDismiss` at `LicenseSettings.tsx:277`/`:453`) — on a successful inline activation the "Done" button calls nothing. It is currently unreachable (entitlement flip unmounts the surface first, per the deferred note), but it is a latent affordance that does nothing if ever reached. **Fix:** either omit the Done button when `onDismiss` is undefined (`{onDismiss ? <button…/> : null}`) or pass a real handler from the pane. Cheap, removes a dead control.

## Detailed Findings

### Pillar 1: Copywriting (4/4)
Excellent. Every string is specific and calm:
- CTAs are concrete: "Buy license", "I have a license key", "Activate", "Refresh", "Deactivate this device" — no "Submit"/"OK"/"Click here" (the only `onSubmit`/`Submit` grep hits are the React handler name and the `ERROR_COPY` map, not UI labels).
- Error copy (`UpsellPanel.tsx:73-84`) is differentiated and actionable: seatLimit names the resolution path, offline vs serviceUnreachable are distinct, invalidKey says "check it and try again". Device-neutral ("device", never "Mac") per the locked walkthrough decision.
- Empty/attention states carry reassurance, not alarm: "Your tools keep working — activate again to restore your license" (`:235`), "Your themes and tool order are saved" (`:233`).
- Honest copy: "Lost your key? Check your purchase email" (`LicenseSettings.tsx:455`) deliberately does NOT link a portal that doesn't exist. The Keychain heads-up (`UpsellPanel.tsx:348-354`) pre-empts the macOS system prompt — proactive, reduces surprise.
- Claims footer "One-time payment · Free updates forever · 14-day refund" — all user-confirmed true per `deferred-items.md`.

### Pillar 2: Visuals (4/4)
Strong, differentiated hierarchy verified against the three rendered screenshots:
- **Free pitch** (`license-states-free.png`): clear focal point — borderless accent medallion → 24px "Thank you for using TinkerDev ❤️" hero → feature list → $9 price block → CTAs. The accent radial glow (`PITCH_GLOW_STYLE`, `:114-117`) draws the eye top-down. This is the single strongest focal composition in the app.
- **Problem** (`license-states-problem.png`): 3-column amber banner [AlertTriangle | content + UNVERIFIED pill | Refresh] reads as a calm warning; the separate "Activate a license" section below is visually distinct.
- **Licensed** (`license-states-licensed.png`): green dot + "Licensed" + PRO pill banner, then a neutral Licensee/key detail table, then a left-aligned Deactivate trigger — clean rank from status → detail → destructive action.
- Every icon is `aria-hidden="true"` and paired with adjacent text (AlertTriangle/Lock/RefreshCw/Key/Check) — no icon-only buttons. Hierarchy comes from size (24/20/16/14/12px), weight (semibold vs normal), and the warn/ok/bad/accent color system, not from any single channel.

### Pillar 3: Color (4/4)
- New tokens are correctly derived: `--color-warn` (#fbbf24) and the `--color-ok` (#34d399) triads each add a 15% `-soft` fill + 75% `-line` via `color-mix`, mirroring the accent triad exactly (`index.css:55-77`) — a future theme re-derives from the base, no orphaned literals.
- Semantic discipline is clean: amber = attention (problem/refreshNeeded), green = Pro-active, red `--color-bad` = destructive deactivate, accent = primary CTA + focus rings ONLY. Accent grep on the changed files: `bg-accent` ×3, `text-accent` ×3, `ring-accent` ×4, `border-accent` ×1 — all on CTAs, medallions, and focus rings, never on status glyphs (matches the D-24 rule in the file header).
- No hardcoded colors in JSX — the only `#hex` grep hits are explanatory contrast-math comments (`UpsellPanel.tsx:123`, `:320`).
- Contrast claims are documented per surface and clear AA: warn ~10.8:1 on bg-pane, ok ~7.1:1 on bg-ok-soft, destructive text-bad ~5.7:1 on the bad-tinted card (`LicenseSettings.tsx:74-78`), placeholder text-tx-3 ~5.6:1. **Not independently re-measured here** (no automated contrast tool run) — recommend the phase-boundary axe pass confirm the tinted-surface trio (warn-soft, ok-soft, bad/10) since `color-mix` over a translucent fill composited on the gradient body can drift from the documented opaque-surface math.
- No opacity-only state encoding: disabled buttons swap to explicit `bg-input-bg`/`text-tx-2` tokens, not just reduced alpha.

### Pillar 4: Typography (3/4)
- One family (IBM Plex Sans) and only 2 weights in the changed surface (`font-semibold` ×9, `font-medium` ×1) — disciplined.
- BUT 9 distinct arbitrary px sizes appear: 10, 11, 12, 13, 14, 16, 20, 24, 28px. The 12px body dominates (26 uses), and the larger steps (24/28px) are justified hero/price moments, but 10px (pills, `:331`/`:383`) and 11px (claims footer, `:520`) are very small — 10px uppercase mono pills lean on the high warn/ok contrast to stay legible. This exceeds the >4-size guideline. Minor: the scale is intentional and each size has a role, so this is a 3 not a 2. **Recommendation:** if a type scale is ever tokenized, collapse 11px→12px (claims footer) and confirm the 10px pills hold AA at that size; today it is acceptable but on the edge.

### Pillar 5: Spacing (4/4)
- Zero arbitrary spacing values (`gap-[…]`/`p-[…]` grep returns nothing) — everything is on the Tailwind scale.
- Consistent rhythm: banners use `p-5`, cards `p-6`, pane padding `p-8`, `gap-2` is the workhorse (15 uses), section gaps `gap-3`/`gap-4`. The amber attention banner, green success banner, and red confirm card all share the `items-start gap-3 … p-5` 3-column structure — deliberate parallelism that makes the states feel like one family.
- Form-narrow clamp (`max-w-[420px]`) is the only arbitrary value and it is a width, not spacing — appropriate (forms read better narrow).

### Pillar 6: Experience Design (3/4)
State coverage is comprehensive:
- **Loading:** `refreshing`/`pending` drive `aria-busy`, `animate-spin` on RefreshCw, and "Refreshing…"/"Activating…" aria-live lines. Input uses `readOnly` (not `disabled`) during submit to avoid dropping focus to `<body>` — a known WKWebView pitfall handled correctly (`UpsellPanel.tsx:313`).
- **Error:** typed `ERROR_COPY`/`REFRESH_ERROR_COPY` maps, calm text-bad rendering, field keeps value + refocuses on failure (`:267-268`).
- **Empty/attention:** notActivated → full upsell; refreshNeeded/problem → status card + inline form; D-84 one-time drop notice.
- **Destructive confirmation:** confirm-first inline Deactivate with proper focus management — focus → confirm control on reveal, → trigger on cancel via the `focusAfter` ref + effect keyed on `confirming` (`:151-168`); offline guidance is server-delete-first and stays calm (never text-bad).
- **aria-live:** 10 regions in LicenseSettings, 5 in UpsellPanel — status-label transitions, in-flight, and error all announce politely.
- 6 `focus-visible:ring` usages in UpsellPanel, 4 in LicenseSettings — every interactive control has a visible focus indicator; real `<label htmlFor>` for the key input (`:289`), `aria-describedby` for the stored-key hint.

**Why 3 not 4:** the intentional h3→h2 heading jump (top fix #1) inverts local heading order for SR users, and the form-only mount-focus (top fix #2) lands SR users below the explanatory status card. Both are documented, deliberate trade-offs — but both are real a11y experience gaps against the WCAG-AA floor this project treats as binding, so the pillar can't be a clean 4.

## Minor Recommendations (non-blocking)

- **Refresh button disabled tokens on the amber card** (`deferred-items.md`): during the brief in-flight `disabled` state the Refresh button renders neutral `bg-input-bg` on the amber banner — not an AA failure (the button carries its own surface), purely transient/aesthetic. Revisit only if a warn-toned button variant is added.
- **Redundant mount status re-query**: `ActivationSurface` fires `refreshLicenseUi()` on mount (`UpsellPanel.tsx:220`) while `LicenseSettings` already fires `refreshLicenseUiDetailed()` (`:106`). Both pure-local for inline variants, no clobber — cheap future fix is to gate the surface's mount-refresh to `variant === "panel"`.
- **Stale stacking comments** in `SettingsModal.tsx`/`App.tsx` still describe the removed License-pane→UpsellModal path — comment-only drift, refresh when next touched.
