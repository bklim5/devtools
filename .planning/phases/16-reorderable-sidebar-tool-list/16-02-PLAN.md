GRIP HANDLE (D-01, REORD-01/02):
- Each row gains a grip handle element (e.g. a lucide `GripVertical` icon button) that is
  `opacity-0` by default and becomes visible on row `group-hover` AND on the handle's own
  `focus-visible` (so keyboard users can find it). The handle is `draggable`, NOT the NavLink.

- Use native HTML5 drag events on the handle/row: `onDragStart` (set the dragged id; e.g.
  `e.dataTransfer.setData('text/plain', id)` and track `draggingId` in state),
  `onDragOver` (preventDefault to allow drop; compute the hovered target index → set
  `dropIndex` state), `onDrop` (compute final index, call
  `setToolOrder(moveToolInOrder(orderedIds, draggingId, targetIndex))`, announce, clear
  state), `onDragEnd` (clear drag state). ZERO new deps — native events only.
- Render a neutral insertion line between rows at `dropIndex` during a drag — a thin
  (~2px) bar using a NEUTRAL token (e.g. `bg-bd` / `bg-tx-2/40`), explicitly NOT

KEYBOARD REORDER (D-04/D-05, REORD-03/04):
- On the handle's `onKeyDown`: handle ONLY `Alt+ArrowUp` and `Alt+ArrowDown` (require
  `e.altKey`). For each: compute `currentIndex = orderedIds.indexOf(id)`, target =
  `currentIndex - 1` (up) or `currentIndex + 1` (down), call
  `setToolOrder(moveToolInOrder(orderedIds, id, target))`, `e.preventDefault()`, announce,
  and RE-FOCUS the same tool's handle after the re-render (the moved item keeps focus —
  e.g. a ref map keyed by tool id, focus in a layout effect when an `idToFocus` marker is set).
- Do NOT bind plain `ArrowUp`/`ArrowDown` (no `e.altKey`) — they must fall through unhandled
  (no roving navigation, D-05). Only Alt+arrow is consumed.

ARIA-LIVE (D-06, REORD-04):
- Add ONE `aria-live="polite"` region (visually hidden, e.g. `sr-only`) inside the aside.
- On every successful move (drag OR keyboard), set its text to exactly:
  `Moved {tool.name} to position {n} of {total}` where `n` is the 1-based new index in the
  reconciled order and `total` is `orderedIds.length`. Use the registry `tool.name` (never
  raw stored input) — this also closes the injection concern (T-16-04).

RESET (D-12, REORD-07):

  `Sidebar order reset to default`. Exact placement/label/markup is Claude's discretion, but
  it MUST be reachable and MUST set toolOrder back to `[]`. Ensure the menu is dismissible
  (click-away / Escape) and does not trap focus.

Keep all Tailwind layout-agnostic (no fixed pixel widths beyond the existing aside width);
honor compact density and the accent=selected rule throughout.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted `toolOrder` → Sidebar render | Untrusted saved order reaches the render/overlay; reconciled before use. |
| tool IDs/names → DOM + aria-live text | Strings flow into rendered markup and the live-region announcement. |
| drag/keyboard events → reorder mutation | User input drives `moveToolInOrder` + `setToolOrder` writes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-05 | Tampering | Sidebar render of a tampered/oversized `toolOrder` | mitigate | Render goes through `reconcileToolOrder` (Plan 01) — output bounded by the registry set, unknown ids dropped, duplicates collapsed; the list can never crash, drop, or duplicate a tool (D-11/REORD-06). |
| T-16-06 | Injection (XSS) | tool name/id into DOM + `aria-live` text | mitigate | React escapes text content by default; the aria-live message uses the registry `tool.name` (controlled), never raw stored `toolOrder` strings. No `dangerouslySetInnerHTML`. |
| T-16-07 | Elevation/Scope creep | drag/keyboard mutating the registry or router | mitigate | Reorder only writes the `toolOrder` pref overlay; `ENABLED_TOOLS`, the ⌘K palette, and the router are never mutated (D-10) — registry stays the single control plane. |
| T-16-08 | Denial of Service | rapid drag/key spam writing prefs | accept | Each move is a single bounded `setToolOrder` (write-on-change, Pitfall-5 pattern); the blob is tiny (≤11 ids). No unbounded growth; local-only, no network. Acceptable. |
</threat_model>
