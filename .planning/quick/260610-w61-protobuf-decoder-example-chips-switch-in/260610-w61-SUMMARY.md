# Quick 260610-w61: Protobuf Example Chips Switch Input Encoding Summary

Example chips now carry their own `InputEncoding` and a chip click sets both `raw` and an explicit `override`, so every example decodes under its matching mode even when the user had previously forced a different encoding via the segmented control.

## What was done

- **Task 1 (`505439e9`)** — `EXAMPLES` extended to `{ label, value, encoding: InputEncoding }` (four hex payloads + the decimal array); chip `onClick` now does `setRaw(ex.value); setOverride(ex.encoding)`. Segmented control needed zero changes — its active segment already renders `result.encoding`, and `override` is already part of `decodeKey` so selection/collapsed reset correctly. Three new unit tests: forced-base64 → `{1:150}` chip flips to hex + decodes (no alert); forced-hex → "decimal bytes" chip flips to decimal + decodes; clicking the now-active segment after a chip click clears the override back to auto-detect with no regression. Existing D-03/D-10 tests pass unmodified.
- **Task 2 (`966fc5d0`)** — new real-WKWebView e2e spec in `test/e2e/protobuf-decoder.e2e.ts`: seed hex, force base64 (genuinely mismatched), click "nested message" → hex segment `aria-pressed="true"`, base64 released, field renders, no alert; then "decimal bytes" → decimal segment active, clean decode. Full e2e gate via `scripts/e2e-spike.sh`: **15/15 spec files**, exit 0; orphan vite/app processes reaped after.
