# Milestone v1.4: Reorderable Tools

**Status:** ✅ SHIPPED 2026-06-05
**Phases:** 16 (single phase)
**Total Plans:** 2

## Overview

A focused single-feature milestone: a **user-reorderable sidebar tool list** — the first milestone to add a *personalization* capability rather than a tool. The user can reorder the sidebar by drag-and-drop (handle-initiated native HTML5 drag, no dnd library) or by an accessible keyboard path (Alt+↑/↓ with `aria-live` announcements), and the custom order persists across restarts as a `toolOrder: string[]` overlay applied over the registry at render time — the registry array stays the single control plane (the ⌘K palette and router remain order-agnostic). Graceful reconciliation handles new/removed tools, and a keyboard-reachable "Reset order" action restores the default.

Promoted from backlog 999.6 (12 locked design decisions gathered 2026-06-04). Inherits the binding wedge: offline/no-network, keyboard-driven, registry-driven single control plane, WCAG-AA, **zero new runtime dependencies**, and `src/lib/protobuf/decoder.ts` + its 19 tests byte-for-byte untouched.

## Phases

### Phase 16: Reorderable Sidebar Tool List

**Goal**: A user can reorder the sidebar tools to suit their own workflow — by drag-and-drop or by keyboard — and that order is remembered across restarts, while the registry stays the canonical source of truth.
**Depends on**: Phase 2 (Shell — `Sidebar.tsx`, registry projection) and Phase 5 (`usePreferences` / `platform.store` persistence seam); both shipped. No intra-milestone dependencies (single phase).
**Requirements**: REORD-01, REORD-02, REORD-03, REORD-04, REORD-05, REORD-06, REORD-07
**Plans**: 2 plans

Plans:

- [x] 16-01: Persistence + pure ordering/reconciliation backbone — `toolOrder: string[]` Preferences field persisted through the existing prefs seam, `coerceToolOrder` untrusted-merge, `setToolOrder` write-on-change setter, and two pure fully-tested helpers (`reconcileToolOrder` D-11 render overlay — always a registry permutation — + `moveToolInOrder` clamped relocate). REORD-05/06/07. 40/40 suite tests, decoder 19/19 untouched, zero new deps. Commits `90857271`, `72955ab3`.
- [x] 16-02: Reorderable Sidebar UI — reconciled `toolOrder` overlay over `ENABLED_TOOLS` (registry/⌘K/router untouched), handle-initiated native HTML5 drag with a neutral (`tx-2`, non-accent) insertion line + end-of-list drop zone, Alt+↑/↓ keyboard reorder (one slot/press, moved item keeps focus, plain arrows unbound), `aria-live="polite"` "Moved {tool} to position N of M" announcements, and a keyboard-reachable Reset-order affordance (right-click + Shift+F10 + focus-on-open + Escape-restore). REORD-01..07. Human-approved on the real WKWebView. Commits `026575b4`, `a3dc2927`, `f91a777a`, `4c64b900`, `8c23ac9a`, `b7896524`.

**Success Criteria** (all TRUE at sign-off):
  1. Grab a tool by its grip handle and drag to a new position — a subtle, neutral (non-accent) insertion line shows where it will land — and a plain click still navigates without ever starting a drag. *(REORD-01, REORD-02)*
  2. Focus a tool's handle and press Alt+↑ / Alt+↓ to move it one slot per press; plain arrow keys stay unbound (no roving navigation), and the moved tool keeps focus. *(REORD-03)*
  3. Every reorder is announced through an `aria-live="polite"` region ("Moved Cron to position 3 of 11") so the change is perceivable without sight. *(REORD-04)*
  4. The custom order survives restart: stored as `toolOrder: string[]` through the existing preferences/store seam, applied as a render-time overlay over `ENABLED_TOOLS` (registry unchanged; palette + router order-agnostic). *(REORD-05)*
  5. On load the saved order is reconciled against the live registry — a later-shipped tool appears at the bottom, an unknown/removed ID is ignored — so the list never crashes, drops, or duplicates a tool; and a "Reset order" action restores the default registry order. *(REORD-06, REORD-07)*

**Details:**

A render-time presentation overlay, never a registry mutation. `Sidebar.tsx` renders `reconcileToolOrder(preferences.toolOrder, ENABLED_TOOLS.map(t => t.id))` mapped through `getToolById`; the registry array, ⌘K palette, and router are never touched. The grip handle (`GripVertical`) appears on row hover AND its own `focus-visible`; only the handle is `draggable`, so a plain `NavLink` click still navigates. Keyboard reorder consumes only Alt+↑/↓ (no roving nav), with a ref-map restoring focus to the moved item after re-render. One visually-hidden `aria-live="polite"` region announces every move using the registry-controlled `tool.name`. Reset is a right-click context menu plus a keyboard-reachable Shift+F10 path (focus-on-open, Escape-restore) that calls `setToolOrder([])`. Persistence rides the Plan-01 prefs seam (`setToolOrder`), surviving a webview reload and a real app restart.

**Two notable deviations** (both auto-fixed, no scope change): the drop-indicator token moved `bd-2`→`tx-2` to clear WCAG 1.4.11 non-text contrast (3:1) while staying neutral; and the WebKit-driver Alt-modifier key-chord gap was handled in the e2e by dispatching a bubbling `KeyboardEvent` to exercise the real `onKeyDown` handler.

## Milestone Summary

**Decimal Phases:** None.

**Key Decisions:**

- **Ordering is a render-time presentation overlay** over `ENABLED_TOOLS`, never a registry mutation — the registry stays the single control plane; ⌘K palette + router stay order-agnostic (D-10).
- **`reconcileToolOrder` output is always a registry permutation** — registry IDs in `toolOrder` render in saved order, registry IDs absent from it append in registry order, `toolOrder` IDs no longer in the registry are dropped, duplicates collapse (D-11) — the list can never crash, drop, or duplicate a tool.
- **Persist via the existing `usePreferences` / `platform.store` seam** — one additive `toolOrder: string[]` field, write-on-change, mirroring `recentToolIds` (but with NO length cap, so `coerceToolOrder` is a dedicated coercer rather than folded into `normalizeRecents`).
- **Drag is handle-initiated** (grip on hover + focus, only the handle `draggable`) so a plain click still navigates (D-01/D-02); **drop indicator is a neutral insertion line** — accent stays selected-only.
- **Keyboard reorder is Alt+↑/↓ only** (one slot per press) — NO roving arrow nav introduced (plain arrows stay unbound, preserving the Phase-2 pointer+Tab-focus model); moved item keeps focus; each move announced (D-04/D-05/D-06).
- **Reset via a keyboard-reachable context menu** (Shift+F10 + focus-on-open + Escape-restore), setting `toolOrder=[]` (D-12).

**Issues Resolved:**

- WCAG 1.4.11 non-text contrast on the neutral drop indicator — `bd-2`→`tx-2`.
- WebKit WebDriver Alt-modifier key-chord gap — e2e drives the real handler via a bubbling `KeyboardEvent`.
- Three gsd-ui-review a11y findings closed (boundary-move announcement, keyboard-reachable reset, explicit end-of-list drop zone).
- **Post-ship drag fix (`1c2c7664`):** mouse drag-to-reorder showed the drag image but never moved rows on the real WKWebView — Tauri v2 window `dragDropEnabled` defaults to `true`, so the OS-level file-drop handler intercepted the webview's HTML5 `dragover`/`drop` (the keyboard path was unaffected — pure DOM). Set `dragDropEnabled:false` in `tauri.conf.json` (safe; the app has no file-drop feature). NOT caught by the gates: WebDriver can't synthesize native OS drag, so the e2e only exercised the keyboard path — the pointer drag rode on the manual walkthrough.

**Issues Deferred:** None blocking.

**Technical Debt Incurred:** None.

**Deferred by design (not this milestone):**

- **Pinning** — locking the Protobuf hero to the top, or pinning favourite tools above the reorderable list (explicitly split out as its own future feature, user decision 2026-06-04).
- **Settings surface** — a dedicated preferences/appearance screen that would give the "Reset order" control (and future appearance settings) a home; none exists today.
