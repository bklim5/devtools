(D-16 promotes `toggleClasses(active)` + the segmented `<button aria-pressed>` markup):

```tsx
function toggleClasses(active: boolean): string {
  return [
    "rounded-[5px] px-2 py-0.5 text-[11px] font-medium outline-none transition-colors",
    "focus-visible:ring-2 focus-visible:ring-accent",
    active
      ? "border border-accent-line bg-accent-soft text-accent"
      : "border border-transparent text-tx-2 hover:text-tx",
  ].join(" ");
}
// segmented group wrapper (from Base64Tool AlphabetToggle):
// <div role="group" aria-label={...} className="flex items-center gap-1 rounded-[7px] border border-bd bg-input-bg p-0.5">
//   <button type="button" aria-pressed={active} className={toggleClasses(active)}>{label}</button>
```

Error-as-value pattern already used in src/lib/bytes.ts (helpers throw a plain named Error;
the caller try/catches). For url.ts the helpers themselves return a discriminated result so
the view never needs its own try/catch — choose ONE consistent shape, e.g.:

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Denial of Service | parseUrl / decode helpers on malformed input | mitigate | Every native call (`new URL`, `decodeURI`, `decodeURIComponent`, `encodeURI`) wrapped in try/catch → error-as-value (D-13/D-14); a thrown URIError can never crash the tool. Covered by url.test.ts %zz + relative-URL cases. |
| T-13-02 | Information Disclosure | password field returned by parseUrl (D-09) | accept | Surfaced plainly by explicit user decision (auth-URL debugging). Mitigation in scope: the helper returns it as a plain string only — never logged, never persisted (no console.log, no platform.store write in this module). Offline app, local-only; user already possesses the URL. |
| T-13-03 | Tampering | rendering decoded values (downstream, 13-02) | mitigate | url.ts returns plain strings; no HTML. The view (13-02) MUST render via React text nodes (default escaping), never dangerouslySetInnerHTML — asserted there. |

No high-severity threats: offline, no network, no auth, no backend, no persistence, no code-eval. The realistic surface is throw-safety (mitigated) and the consciously-accepted plaintext password display.
