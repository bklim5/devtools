Implement the next-run engine and wire it into `analyzeCron`. Land tests GREEN in the same commit.

1. **`nextRuns(f: CronFields, now: Date, zone: string, want = 5): CronRun[]`** — day-granular wall-clock odometer (RESEARCH §"Next-run odometer"):
   - Define a hard cap constant `const CANDIDATE_DAY_CAP = 5 * 366;` (~5 years; Assumption A2). This is the freeze protection (CRON-08) — NEVER a `while(true)` without it.
   - Start from the zoned date of `now`. Walk candidate days from today forward, at most `CANDIDATE_DAY_CAP` iterations, advancing one day via `Date.UTC` overflow normalization (`addOneDay`).
   - For each candidate day: skip if `!f.month.has(month)`; skip if `!dayMatches(f, y, mo, d)`.
   - For a matching day, iterate `hour` × `minute` × `second` in ASCENDING sorted order; for each, `const inst = wallClockToInstant(y, mo, d, h, mi, s, zone)`; if `inst !== null` and `inst.getTime() > now.getTime()`, it's a candidate run.
   - **Fall-back de-dupe (CRON-07/Pitfall 6):** track the last emitted instant's epoch ms; skip an instant whose getTime() equals the previously emitted one so a repeated wall-clock hour yields ONE run.
   - **Spring-forward (CRON-07):** `wallClockToInstant` returning `null` naturally skips the non-existent time (the locked "skip + advance" convention, CONTEXT decision).
   - Stop once `want` (5) runs collected.
   - Build each `CronRun`: `date: inst`, `label:` a 24-hour LOCAL string via `new Intl.DateTimeFormat(undefined, { dateStyle:"medium", timeStyle:"short", hourCycle:"h23", timeZone: zone }).format(inst)` (24-hour so midnight never renders 24:00; the view adds the relative caption + zone caption separately).

2. **Wire into `analyzeCron`:** after parsing valid fields, call `const runs = nextRuns(fields, now, zone, 5)`. If `runs.length === 0` → return `{ kind: "never", description: describe(fields, sixField) }` (CRON-08: the description still renders; the expression parsed fine, it just never fires within the cap). Otherwise return `{ kind: "scheduled", description, runs }`. The system zone is supplied by the caller (the view passes `Intl.DateTimeFormat().resolvedOptions().timeZone`); analyzeCron's `zone` param is that value — do NOT call resolvedOptions inside the pure core (keep it injectable for fixtures).
