```typescript
export interface CronRun { date: Date; label: string; }
export type CronResult =
  | { kind: "scheduled"; description: string; runs: CronRun[] }
  | { kind: "reboot"; description: string }
  | { kind: "never"; description: string }
  | { kind: "empty" }
  | { kind: "error"; message: string };

// Normalized field model — parse once, match many.
export interface CronFields {
  second: Set<number>;   // 0..59  (6-field only; defaults to {0} for 5-field)
  minute: Set<number>;   // 0..59
  hour: Set<number>;     // 0..23
  dom: { values: Set<number>; restricted: boolean; lastDay: boolean; lastOffset?: number }; // 1..31 + L/L-n (markers parsed in Plan 03)
  month: Set<number>;    // 1..12
  dow: { values: Set<number>; restricted: boolean; lastWeekday?: number };  // 0..6 (7→0) + nL (Plan 03)
}
export function analyzeCron(input: string, now: Date, zone: string): CronResult;
export function describe(f: CronFields, sixField: boolean): string;
```

Implement `analyzeCron(input, now, zone)` parse front-half (next-run computation is Plan 02 — for THIS plan, return `{ kind: "scheduled", description, runs: [] }` for valid non-reboot expressions; Plan 02 fills `runs`). Pipeline:

1. **Empty:** `input.trim() === ""` → `{ kind: "empty" }`.

2. **Macro expansion** (case-insensitive, CRON-03). Trim, lowercase for the macro check. Mapping (expand to a 5-field string, then continue parsing):
   - `@yearly` / `@annually` → `0 0 1 1 *`
   - `@monthly` → `0 0 1 * *`
   - `@weekly` → `0 0 * * 0`
   - `@daily` / `@midnight` → `0 0 * * *`
   - `@hourly` → `0 * * * *`
   - `@reboot` → SENTINEL: return `{ kind: "reboot", description: "At startup — runs once when the scheduler starts." }` (CRON-09 — NO field parse, NO runs).
   - Any other `@token` → `{ kind: "error", message: 'Unknown macro "<token>" — try @yearly, @monthly, @weekly, @daily, @hourly, or @reboot.' }`

3. **Unsupported-token reject (CRON-F1, locked):** if the raw expression contains `W` (word-boundary-insensitive — match the chars), `#`, or `LW`, return `{ kind: "error", message: "`W`, `#` and `LW` aren't supported yet — try a standard field, range, step, list, or `L` / `nL` / `L-n`." }`. (Note: bare `L` and `nL`/`L-n` are SUPPORTED — Plan 03 — so only reject `W`, `#`, and the literal `LW`. Detect `LW` before the bare-`L` allowance.)

4. **Field-count disambiguation (CRON-02, Pitfall 4):** split the (macro-expanded) expression on whitespace, drop empty tokens. Count:
   - 5 tokens → `[minute hour dom month dow]`, with `second` defaulting to `new Set([0])`.
   - 6 tokens → `[second minute hour dom month dow]`.
   - any other count → `{ kind: "error", message: 'Wrong field count: expected 5 or 6 fields, got <n>.' }`

5. **Per-field parse** via a helper `parseField(token, min, max, names?)` returning either a `Set<number>` (for numeric fields) or `{ error }`. Pipeline per field: split top-level on `,` (lists) → each item is one of `*` | `<n>` | `<a>-<b>` | `<base>/<step>` | a name → expand → union → range-check every produced value against [min,max]. Concrete rules:
   - `*` → full range [min..max] (mark field as NOT restricted).
   - `<base>/<step>` where base is `*`→[min..max], `a-b`→[a..b], bare `a`→[a..max]; then keep every step-th element from the range's low end (Pitfall 2: `0-30/10`→{0,10,20,30}, `*/15` minute→{0,15,30,45}). Step must be ≥1 else error.
   - `<a>-<b>` → inclusive [a..b] (Pitfall 7); require a≤b else `{ error: 'Invalid range "<a>-<b>": start must be ≤ end.' }`.
   - `<n>` → {n}.
   - names: month JAN..DEC (1..12, case-insensitive); dow SUN..SAT (0..6, case-insensitive). Map names BEFORE numeric parse.
   - any out-of-range value → `{ kind: "error", message: 'Out of range: <fieldName> "<value>" must be <min>–<max>.' }` (name the bad token AND the valid range, mirroring url.ts/bytes.ts named errors — the UI-SPEC copy contract).
   - unparseable item → `{ kind: "error", message: 'Unparseable token: "<item>" in <fieldName>.' }`.

6. **Field ranges & names** (CONTEXT/RESEARCH §field grammar):
   - second 0–59, minute 0–59, hour 0–23, day-of-month 1–31, month 1–12 (names JAN..DEC), day-of-week 0–7 with names SUN..SAT.
   - **DOW normalization (Pitfall 3):** parse dow against 0–7, then normalize `7→0` in the resulting Set (so `0` and `7` both = Sunday — CRON-06). Keep month 1-based internally.

7. **`restricted` flag (for Plan 02's OR-union):** for dom and dow, set `restricted = true` iff the field was an explicit list/range/single/step-over-a-subrange — i.e. NOT a bare `*` and NOT `*/n` over the full range. A bare `*` and `*/n` (full-range step) both count as NOT restricted. Store this on `dom.restricted` / `dow.restricted`.

8. For dom/dow this plan: set `lastDay: false`, `lastOffset` unset, `lastWeekday` unset (Plan 03 fills the L-markers — leave the struct shape ready). `second/minute/hour/month` are bare `Set<number>`.

Return `{ kind: "scheduled", description: describe(fields, sixField), runs: [] }` for valid expressions (Plan 02 replaces `runs: []` with the real next-run computation).

TESTS (`cron.test.ts`, GREEN with the impl): assert the parsed Set contents for: `0 9 * * 1-5`, `*/15 * * * *`, `0-30/10 * * * *` (→{0,10,20,30}), `0 0 1 JAN *`, `* * * * 0` and `* * * * 7` (both → dow contains 0), 6-field `30 0 9 * * *` (second={30}). Assert error kinds + exact messages for: 4-field input, `0 99 * * *` (minute out of range), `0 0 * * MONDAY`-style unparseable, `@bogus`, and `0 0 * * 1W` / `0 0 1#1 * *` (W/# reject). Assert `@reboot`→kind:"reboot". Assert `""`→kind:"empty".
