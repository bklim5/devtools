# Milestone v1.5: Pinned Tools

**Status:** 🚧 IN PROGRESS — started 2026-06-05
**Phases:** 17 (single phase, continues from v1.4's Phase 16)
**Total Plans:** 2

## Overview

A focused single-feature milestone that extends v1.4's personalization: users **pin favourite tools to a distinct, reorderable "Pinned" section at the top of the sidebar**, so the tools they reach for most sit above the rest — independent of the v1.4 custom order. Pinning is a **render-time presentation overlay** persisted as a `pinnedToolIds: string[]` field through the existing `usePreferences`/`platform.store` prefs seam (beside `toolOrder`/`recentToolIds`), reconciled against the live registry on load. The registry stays the single control plane — the ⌘K palette and router remain pin-agnostic.

Promoted from the v1.4 carry-forward "Pinning" future requirement (split out 2026-06-04; design confirmed via `/gsd-new-milestone` questioning 2026-06-05). It is a near-clone of v1.4 in structure — an overlay persisted through the prefs seam plus a sidebar UI — and reuses v1.4's machinery directly: the `reconcileToolOrder`/`moveToolInOrder` ordering helpers, the grip-handle native HTML5 drag + Alt+↑/↓ keyboard reorder, and the `aria-live` announcement pattern.

Inherits the binding wedge: offline/no-network · keyboard-driven · registry-driven single control plane · HashRouter only · WCAG-AA (the keyboard path + `aria-live` are mandatory, not optional) · layout-agnostic · **zero new runtime AND dev dependencies** · `src/lib/protobuf/decoder.ts` + its 19 tests byte-for-byte untouched · macOS real-WKWebView UI gate. **Native-OS drag is manual-walkthrough coverage** — WebDriver can't synthesize it (`dragDropEnabled:false` already set from v1.4).

**Assumed defaults (confirmed, do not re-litigate):** no tool pinned by default (empty pinned section → no divider; the hero is NOT auto-pinned); pinning appends to the bottom of the pinned section; the dedicated settings surface stays deferred.

## Phases

### Phase 17: Pinned Sidebar Section

**Goal**: A user can pin the tools they use most into a distinct "Pinned" section above the rest of the sidebar — by a row pin icon or a keyboard shortcut — and that pinned set is remembered across restarts, reorderable independently of the unpinned list, while the registry stays the canonical source of truth.
**Depends on**: Phase 2 (Shell — `Sidebar.tsx`, registry projection), Phase 5 (`usePreferences`/`platform.store` persistence seam), and Phase 16 (the `toolOrder` overlay, `reconcileToolOrder`/`moveToolInOrder` helpers, drag + Alt+↑/↓ + `aria-live` patterns this milestone reuses) — all shipped. No intra-milestone dependencies (single phase).
**Requirements**: PIN-01, PIN-02, PIN-03, PIN-04, PIN-05, PIN-06, PIN-07, PIN-08, PIN-09
**Plans**: 2 plans (mirrors v1.4's 16-01 backbone / 16-02 UI split)
**UI hint**: yes

Plans:

- [ ] 17-01: Persistence + pure pinning backbone — an additive `pinnedToolIds: string[]` Preferences field persisted through the existing prefs seam (`coercePinnedToolIds` untrusted-merge — string-only, de-dupe, non-array → `[]`, mirroring `coerceToolOrder`; `setPinnedToolIds` write-on-change setter; a `togglePinned` add/remove helper), plus the pure, fully-tested partition backbone: coerce + reconcile the saved pinned set against the live registry (drop unknown/removed IDs, collapse duplicates — never crash/drop/duplicate) and split the reconciled tool list into an ordered **pinned group** + ordered **unpinned remainder** for render, reusing `reconcileToolOrder`/`moveToolInOrder` for the per-group order. **REORD-style immovable bar:** output is always a registry partition (every registry tool appears exactly once across the two groups). *(PIN-07, PIN-08; pure backbone for PIN-01/02/06)*
- [ ] 17-02: Pinned Sidebar UI — `Sidebar.tsx` renders the **pinned group** (with a neutral divider, shown only when non-empty) above the **unpinned list**, both projected from the Plan 17-01 partition over `ENABLED_TOOLS` (registry array, ⌘K palette, and router never mutated). A **pin icon** on row hover + `focus-visible` (mirroring the v1.4 grip handle, neutral tokens — accent stays selected-only) toggles membership on click; a **dedicated keyboard shortcut** toggles the currently focused tool with no mouse; both announce through the existing visually-hidden `aria-live="polite"` region ("Pinned {tool}" / "Unpinned {tool}" using the registry-controlled `tool.name`). The v1.4 drag + Alt+↑/↓ reorder runs **independently within each group** — a tool changes membership only by pinning/unpinning, never by dragging across the pinned↔unpinned boundary. A keyboard-reachable **"Unpin all"** sits alongside the existing "Reset order" (calls `setPinnedToolIds([])`). *(PIN-01, PIN-02, PIN-03, PIN-04, PIN-05, PIN-06, PIN-09)*

**Success Criteria** (all TRUE at sign-off — observable on the real WKWebView):
  1. Clicking a tool row's pin icon (visible on hover AND when the row is keyboard-focused, mirroring the v1.4 grip handle) moves that tool into a distinct "Pinned" section at the top of the sidebar; clicking it again on a pinned tool returns it to the main list. *(PIN-01, PIN-02, PIN-04)*
  2. The "Pinned" section and its divider appear only when at least one tool is pinned, and disappear entirely (no empty group, no stray divider) when the last tool is unpinned. *(PIN-03)*
  3. A dedicated keyboard shortcut pins/unpins the currently keyboard-focused tool with no mouse, and every pin/unpin (icon or shortcut) is announced through an `aria-live` region ("Pinned Hash" / "Unpinned Hash"). *(PIN-05)*
  4. Within the pinned group and within the unpinned list, tools reorder independently via the v1.4 drag + Alt+↑/↓ path; a tool changes membership only by pinning/unpinning — dragging never carries it across the pinned↔unpinned boundary. *(PIN-06)*
  5. The pinned set survives an app restart (stored as `pinnedToolIds: string[]` through the existing prefs seam beside `toolOrder`), and a pinned set referencing an unknown/removed tool ID degrades gracefully on load — unknown IDs dropped, duplicates collapsed, no crash and no missing/duplicated tool. *(PIN-07, PIN-08)*
  6. A keyboard-reachable "Unpin all" control (alongside the existing "Reset order") clears the entire pinned set in one action, collapsing the section back to the flat list. *(PIN-09)*

**Details:**

A render-time presentation overlay, never a registry mutation — exactly the v1.4 model, extended to a two-group partition. The reconciled tool list is partitioned into a pinned group and an unpinned remainder; both groups keep the v1.4 per-group order (`reconcileToolOrder` honoring the saved `toolOrder` within each group, `moveToolInOrder` for reorders), and the partition guarantees every registry tool renders exactly once. The registry array, ⌘K palette, and router are never touched (D-10 carried from v1.4). The pin icon mirrors the grip handle's affordance (hover + `focus-visible`, neutral tokens) so a plain `NavLink` click still navigates and accent stays selected-only. The keyboard pin/unpin shortcut and the `aria-live` announcement reuse the v1.4 patterns. Persistence rides the Plan 17-01 prefs seam (`setPinnedToolIds`), surviving a webview reload and a real app restart; `coercePinnedToolIds` makes a malformed/stale stored blob safe (string-only, de-dupe, non-array → `[]`), and reconciliation against the live registry on load drops unknown/removed IDs.

**Verification notes:**
- **Native drag is a manual-walkthrough item** — WebDriver can't synthesize native OS drag (see v1.4 post-ship fix `1c2c7664`; `dragDropEnabled:false` already set). The e2e exercises the keyboard path (pin/unpin shortcut, Alt+↑/↓ within each group, "Unpin all"); the pointer drag-within-group rides the human walkthrough.
- Per-task DoD (in order): simplify → /codex:review → unit green (vitest + tsc + eslint) → real-WKWebView UI verification. Phase boundary: human sign-off on a fresh `tauri build` + a passing `gsd-ui-review` WCAG-AA audit.

## Milestone Summary

**Decimal Phases:** None.

**Key Decisions (carried / extended from v1.4):**

- **Pinning is a render-time presentation overlay** over `ENABLED_TOOLS`, never a registry mutation — the registry stays the single control plane; ⌘K palette + router stay pin-agnostic (extends D-10).
- **The pinned/unpinned partition is always a registry partition** — every registry tool renders in exactly one of the two groups; unknown/removed pinned IDs drop, duplicates collapse (extends D-11) — the list can never crash, drop, or duplicate a tool.

- **Pin affordance mirrors the v1.4 grip handle** — pin icon on hover + `focus-visible`, neutral tokens (accent stays selected-only); a plain `NavLink` click still navigates.
- **Pin/unpin keyboard shortcut + `aria-live` announcement** reuse the v1.4 keyboard + announcement patterns ("Pinned {tool}" / "Unpinned {tool}").
- **Membership changes only via pin/unpin, never drag-across-boundary** — drag + Alt+↑/↓ reorder runs independently within each group; the pinned↔unpinned boundary is fixed during drag.
- **No tool pinned by default** — empty pinned section ⇒ no divider; the hero is NOT auto-pinned (auto-pin/lock-hero deferred).
- **"Unpin all"** sits alongside the existing "Reset order" (the dedicated settings surface stays deferred).
