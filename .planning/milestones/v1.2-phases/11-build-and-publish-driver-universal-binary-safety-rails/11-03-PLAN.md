<objective>
Prove the build-and-publish driver works LIVE on real hardware — the load-bearing milestone acceptance. This plan runs the real (non-dry) `pnpm release:publish`, then exercises the DST-02 updater round-trip: an older install must detect the new release, pass the mandatory minisign verify against the pinned `tauri.conf.json` pubkey, and relaunch into the new version. The universal dual-key behavior (Intel + Apple Silicon served from one artifact) must be confirmed live, not just unit-asserted.

This plan is NOT autonomous — it contains the irreversible publish and the human-only updater round-trip. The driver (Plan 02) does all the automation; this plan is the human gate that confirms it.

Purpose: A broken or unverifiable release auto-installing onto every user is the single worst failure this milestone exists to prevent; the live round-trip is the only proof it doesn't.
Output: `.planning/phases/11-03-SUMMARY.md` recording the publish + round-trip evidence.
</objective>

The driver itself runs every preflight (signing env, rustup both-targets, gh perm, release-not-already-published, tests-green) and aborts before the slow build on any failure.</what-built>
  <how-to-verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintainer shell -> tauri build | Signing key/password live only in the human's exported env; never in the repo, a log, or a CLI arg |
| published latest.json -> every install's updater | The irreversible trust boundary: whatever is served becomes the auto-update every user applies |
| new payload -> minisign verify | The pinned-pubkey verify is the last line of defense; a mismatch must HALT the update, not warn-and-apply |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-13 | Information disclosure | signing env export | mitigate | Human exports the secret into their own shell only; the resume-signal explicitly says do NOT share values; presence-checked without printing |
| T-11-14 | Tampering | live updater payload | mitigate | The mandatory minisign verify against the committed `tauri.conf.json` pubkey is exercised LIVE (Task 3) — a wrong/stale signature fails the round-trip and blocks the publish from being accepted |
| T-11-15 | DoS | dual-arch coverage | mitigate | Round-trip exercised on both arches where possible; single-arch fallback documented as covered-by-construction (one universal artifact, identical signature in both keys — Phase 9 guarantee) |
| T-11-16 | Tampering | wrong-repo publish | mitigate | Task 2 independently confirms the release landed on the PUBLIC `bklim5/devtools-releases` (the endpoint the updater actually polls), never the private `origin` |
</threat_model>
