# Requirements (ARCHIVED): v1.5 Pinned Tools

> **Archived 2026-06-07 ‚Äî milestone SHIPPED.** All 9 PIN requirements validated on the real WKWebView and human-signed-off (Phase 17, single phase). Full suite 694/694; decoder 19/19 untouched; zero new runtime/dev deps; `gsd-ui-review` WCAG-AA 23/24 (0 failures). See `v1.5-ROADMAP.md` and `MILESTONES.md`.

**Milestone goal:** Let users pin favourite tools to a distinct, reorderable "Pinned" section at the top of the sidebar ‚Äî so the tools they reach for most sit above the rest, independent of the v1.4 custom order.

**Source:** Promoted from the v1.4 carry-forward "Pinning" future requirement (split out 2026-06-04). Design confirmed via `/gsd-new-milestone` questioning, 2026-06-05.

**Binding constraints (inherited wedge):** offline/no-network ¬∑ keyboard-driven ¬∑ **registry is the single control plane** (pinning is a render-time presentation overlay; ‚åòK palette + router stay pin-agnostic) ¬∑ WCAG-AA (keyboard path + `aria-live` mandatory) ¬∑ **zero new runtime AND dev dependencies** ¬∑ `src/lib/protobuf/decoder.ts` + its 19 tests stay byte-for-byte untouched ¬∑ macOS real-WKWebView UI gate. **Native-OS drag is manual-walkthrough coverage** ‚Äî WebDriver can't synthesize it (see v1.4 post-ship fix; `dragDropEnabled:false` already set).

**Builds on v1.4:** reuses the `toolOrder`/`recentToolIds` prefs seam, `reconcileToolOrder`/`moveToolInOrder` ordering helpers, the grip-handle drag + Alt+‚Üë/‚Üì keyboard reorder, and the `aria-live` announcement pattern.

- [x] **PIN-02**: User can unpin a pinned tool, returning it to the main (unpinned) list. ‚Äî ‚úì Plan 17-02
- [x] **PIN-03**: The "Pinned" section is shown only when at least one tool is pinned, with a clear divider separating it from the rest of the list. ‚Äî ‚úì Plan 17-02
- [x] **PIN-04**: User can pin/unpin a tool via a pin icon on its sidebar row, visible on hover and when the row is keyboard-focused (mirroring the v1.4 grip handle). ‚Äî ‚úì Plan 17-02 (persistent filled on pinned rows; outline on hover + focus-visible for unpinned)

- [x] **PIN-06**: User can reorder tools within the pinned section and within the unpinned list independently (drag + Alt+‚Üë/‚Üì from v1.4); a tool changes membership only by pinning/unpinning, never by dragging across the pinned‚Üîunpinned boundary. ‚Äî ‚úì Plan 17-02 (keyboard path e2e-proven; native pointer drag is phase-gate manual walkthrough)

- [x] **PIN-09**: User can unpin all tools at once via a keyboard-reachable "Unpin all" control (alongside the existing "Reset order"). ‚Äî ‚úì Plan 17-02 (second item in the Shift+F10 reset menu)

## Outcomes & post-sign-off evolutions (D-17)

Live macOS walkthrough surfaced two issues, fixed before sign-off and recorded as **D-17** (supersedes the planned D-05 "no roving nav"):

- **PIN-05 (Alt+P) macOS fix:** Option+P composes to the character "œÄ", so the original `e.key === "p"` check was dead on the real WKWebView (the first e2e gave a false positive by synthesizing `key:'p'`). Now matches the **physical key** `e.code === "KeyP"`; the e2e dispatches the real `key:'œÄ'/code:'KeyP'` shape as a genuine regression test.
- **PIN-04/06 keyboard model:** the sidebar moved to a Tab-friendly model ‚Äî every row is a Tab stop, the pin button is also Tab-reachable (Enter/Space pins, a keyboard fallback), and plain ‚Üë/‚Üì + Home/End move focus tool-to-tool across the pinned‚Üîunpinned divider (pure `resolveRovingTarget` helper, +9 unit tests). Alt+‚Üë/‚Üì reorder stays clamped within each group (PIN-06 boundary intact). Pin+grip widened to 24√ó24 (WCAG 2.5.8).

These refine *how* PIN-04/05/06 are met; the requirement intents are unchanged.

## Future Requirements (deferred ‚Äî carried forward, not promised)

- **Settings surface** ‚Äî a dedicated preferences/appearance screen that would give "Reset order", "Unpin all", theme, and accent a single home. None exists today; explicitly deferred (user decision, 2026-06-05).
- **Auto-pin / lock the hero** ‚Äî automatically pinning (or locking) the Protobuf hero to the top by default. Not in this milestone (default is no tool pinned).

## Out of Scope (held)

- **Pinning affecting the ‚åòK command palette or routing** ‚Äî only the sidebar render order changes; palette and router stay pin-/order-agnostic.
- **Dragging tools across the pinned‚Üîunpinned boundary** ‚Äî membership changes via pin/unpin only; the boundary is fixed during drag.
- **A dnd/animation library** ‚Äî zero-new-runtime-deps held; native drag events only (as in v1.4).
- **Cross-device pin sync** ‚Äî local-only, offline.
- **A dedicated settings surface** ‚Äî deferred (see Future Requirements).

> **Note ‚Äî deliberate evolution of a v1.4 out-of-scope line:** v1.4 held "no grouping/sections in the sidebar (list stays flat)". v1.5 intentionally introduces exactly one section (the "Pinned" group above the rest). This is the scoped exception, not a general grouping/folders feature.
